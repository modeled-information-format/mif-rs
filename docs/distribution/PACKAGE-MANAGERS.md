# How to Install and Distribute mif-rs

Install `mif-rs`'s libraries or binaries, or understand the three channels
its release pipeline publishes to.

## Prerequisites

- For crates.io/Homebrew installs: nothing beyond `cargo`/`brew` themselves.
- For understanding the distribution pipeline: familiarity with
  [RELEASING.md](../runbooks/RELEASING.md).

## Install the libraries

```bash
cargo add mif-core mif-schema mif-ontology
```

## Install the binaries

### Via cargo

```bash
cargo install mif-cli --locked
cargo install mif-mcp --locked
```

### Via Homebrew

```bash
brew tap modeled-information-format/tap
brew install mif-cli
brew install mif-mcp
```

Each binary gets its own formula (`Formula/mif-cli.rb`, `Formula/mif-mcp.rb`)
in the tap, generated by `.github/workflows/package-homebrew.yml` — one
matrix cell per `[[bin]]` target, resolved from `cargo metadata` rather than
a hardcoded name.

### From a GitHub Release

```bash
gh release download v0.1.0 --repo modeled-information-format/mif-rs \
  --pattern 'mif-cli-0.1.0-linux-amd64'
gh attestation verify mif-cli-0.1.0-linux-amd64 \
  --repo modeled-information-format/mif-rs
chmod +x mif-cli-0.1.0-linux-amd64
```

Every release attaches attested binaries named `{bin}-{version}-{platform}`
for both `mif-cli` and `mif-mcp`, across `linux-amd64`, `linux-arm64`,
`macos-amd64`, `macos-arm64`, `windows-amd64` — plus one combined CycloneDX
SBOM and one `mif-rs-{version}-checksums.txt`. See
[SECURITY.md](../../SECURITY.md#verifying-release-artifacts) for the full
verification reference.

## Trigger a release manually

```bash
# Preview a Homebrew formula without pushing
gh workflow run package-homebrew.yml -f version=0.1.0 -f dry_run=true

# Dry-run the publish chain from a branch (tag-gated steps are skipped)
gh workflow run publish.yml
```

## Troubleshooting

### crates.io publish fails with "No Trusted Publishing config found"

Complete the one-time setup per crate: crates.io crate Settings > Trusted
Publishing > add this repo, workflow `publish.yml`, environment `release`
(not `copilot` — that was the upstream template's environment name before
this repo renamed it). Each of the 5 crates needs this linkage
independently.

### crates.io publish fails with "crate already exists"

Versions are immutable; a duplicate attempt after a successful publish is
benign. `publish.yml`'s guard step already checks each crate against the
registry before attempting to publish it, so a re-run only publishes
crates that are genuinely still missing.

### Homebrew formula push fails

- `HOMEBREW_TAP_TOKEN` missing, expired, or lacking write access to the tap.
- Tap repository does not exist — create it or set the `HOMEBREW_TAP_REPO`
  repository variable.
- In dry-run mode the formula is printed to the job log and nothing is
  pushed.

### Homebrew workflow did not run

The `workflow_run` trigger only proceeds for successful, tag-triggered
Release runs (bot-authored release events don't trigger workflows, so
`workflow_run` is the reliable path). Check the Release workflow's
conclusion, then fall back to manual dispatch as shown above.
